# ADR-001: 3層アーキテクチャの採用

## コンテキスト

Spring Boot アプリケーションの設計において、コードの責務を明確に分離する構造が必要である。
ビジネスロジック・Web層・データアクセス層が混在すると、テストの困難さや変更コストの増大につながる。

## 決定

Controller → Service → Repository の3層アーキテクチャを採用する。

| 層 | クラス | 責務 |
|---|---|---|
| Web層 | `*Controller` | HTTPリクエストの受付・レスポンスの返却 |
| ビジネス層 | `*Service` | ビジネスロジックの実装 |
| データアクセス層 | `*Repository` | データの永続化・取得 |

各層の依存方向は上位から下位への一方向とし、Controller が Repository を直接参照することは禁止する。

## 結果

- **メリット**: 各層の責務が明確になり、単体テストが容易になる。層ごとにモック差し替えが可能。
- **デメリット**: 単純なCRUD処理でも3クラスを作成する必要があり、初期コストがかかる。
- **制約**: Controller から Repository を直接呼び出すことは禁止する。
