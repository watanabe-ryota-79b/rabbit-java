# ADR-002: ドメイン単位のパッケージ構成

## コンテキスト

パッケージ構成の方針として、技術レイヤー単位（`controller/`, `service/` など）とドメイン単位（`user/`, `table/` など）の2つのアプローチがある。
技術レイヤー単位では機能追加のたびに複数のパッケージをまたいだ変更が発生し、関連クラスが分散しやすい。

## 決定

ドメイン単位でパッケージを作成し、その配下に技術レイヤーのサブパッケージを置く構成を採用する。

```
com.rabbit.rabbit_java/
  {domain}/
    controller/
    service/
    repository/
    dto/
    entity/
```

## 結果

- **メリット**: 1ドメインに関連するクラスが同一パッケージ以下に集約され、機能単位での把握・変更が容易になる。
- **デメリット**: 複数ドメインをまたぐ共通処理の置き場（`common/` など）を別途定義する必要がある。
- **制約**: 1ドメインのクラスが他ドメインの内部クラス（`dto`, `entity` など）を直接参照することは避け、必要な場合は共通パッケージへの切り出しを検討する。
